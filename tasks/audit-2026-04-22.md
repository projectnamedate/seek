## SECTION 1 — CRITICAL (mainnet blockers, must fix before any real-money deploy)

### C-1. Wrong SKR mint hardcoded in contract + WRONG DECIMALS
**File:** `contracts/programs/seek-protocol/src/lib.rs:9,15-17`
**Current:** `pub const SKR_MINT: Pubkey = pubkey!("u3BkoKjVYYPt24Dto1VPwAzqeQg9ffaxnCVhTAYbAFF")` — devnet test mint with 9 decimals (assumed).
**Verified mainnet state (queried 2026-04-22):**
- Address: `SKRbvo6Gf7GondiT3BbTfuRDPqLWei4j2Qy2NPGZhW3`
- Decimals: **6** (NOT 9 as contract assumes)
- Supply: 10,238,361,594.78 SKR
- Mint authority: `FMNn5sorEBbEoGQGrh7y3xSbYGt116F12FpL2VTsohiw` (owned by program `SKRiHLtLyB8bbhcJ5HBPYMiLh9GcFLdPaSwozqLteha` — Solana Mobile staking program)
- Market price: ~$0.017 per SKR, $91M market cap
- This is the OFFICIAL Solana Mobile ecosystem token, airdropped to Seeker phone owners Jan 2026. Seek does not control the mint.

**Impact:** If program is deployed to mainnet as-is:
1. Every `#[account(address = SKR_MINT)]` check rejects the real SKR token. Protocol unusable.
2. Even after swapping the mint address, `TIER_1_ENTRY = 1_000_000_000_000` (10^12) would be 1,000,000 SKR on mainnet (6 decimals, 10^6 per SKR), not 1000 SKR. All tier entries would be **1000× too large**. A single tier-1 bet = 1M SKR = ~$17,000 real dollars.
3. Log formatting divides by `1_000_000_000` (10^9). Would print garbage on mainnet.

**Fix:**
- Line 9: swap to mainnet mint address.
- Lines 15-17: change from 10^12/10^9 base to 10^6 base. New values: `TIER_1_ENTRY = 1_000_000_000` (1000 × 10^6), `TIER_2_ENTRY = 2_000_000_000`, `TIER_3_ENTRY = 3_000_000_000`.
- All `msg!` divisors: change `/ 1_000_000_000` to `/ 1_000_000`.
- Backend `backend/.env.example:17`: new mint.
- Mobile `mobile/src/config/index.ts:70-71`: new mint + `DECIMALS: 6`.
- Mobile `fetchRealBalance` already uses `TOKEN.DECIMALS` so will Just Work once updated.
- Backend `solana.service.ts:401` `formatSkr`: change divisor.
- Contract tests need updating.
- Consider `#[cfg(feature = "mainnet")]` vs `#[cfg(feature = "devnet")]` for dual-build compatibility, or just split into a `constants.rs` with feature flags.

**Economic implications of real SKR:**
- Tier 1 entry at $0.017/SKR = $17.27
- Tier 2 entry = $34.54
- Tier 3 entry = $51.81
- Tier 3 win (3x = entry back + 2x profit) pays ~$155.44
- House vault starter at 10M SKR = ~$170,000 real capital at launch
- Expected value per bet at 40% success rate: 0.4×(2x) − 0.6×(1x) = +0.2x EV for player. House edge comes from AI rejecting valid captures (tightly calibrated) or from 60% loss rate.
- Price volatility: if SKR doubles in price, the house's SKR-denominated balance doubles in USD terms but payouts also double. Net neutral to protocol, but means user-facing tier prices change with SKR price.

**Regulatory/policy implications:** Using a listed, tradable token (Upbit, OKX, Solflare) for a skill-game with economic stakes may trigger gambling regulations in some jurisdictions. Age gate is present but may need geo-restrictions (e.g. block US states where skill gaming with prizes is restricted — Washington, Arkansas, etc.) before scaling marketing.

### C-2. Predictable jackpot randomness
**File:** `contracts/programs/seek-protocol/src/lib.rs:682–687`
**Current:** `(slot + timestamp) % 500` → jackpot roll. Both `slot` and `timestamp` are derivable from `Clock::get()` by anyone. A sophisticated player or validator can time `finalize_bounty` transactions to land on slots where the roll == 0, draining the entire Singularity jackpot pool. With ~400ms slot times, that's a trivial precompute.
**Impact:** CRITICAL. Entire jackpot pool is drainable by a single adversary. The whole 20%-of-losses accrual is worthless.
**Fix:** Integrate Switchboard On-Demand VRF (~0.002 SOL per request). Requires splitting `finalize_bounty` into `request_jackpot_roll` (requests VRF) + `consume_jackpot_roll` (callback with randomness). Alternative: Orao VRF. Negligible cost at expected volume (even 10k rolls/day = 20 SOL/day). Listed in README as pre-mainnet. Still outstanding.

### C-3. Challenge period hardcoded to 10s
**File:** `contracts/programs/seek-protocol/src/lib.rs:33`
**Current:** `pub const CHALLENGE_PERIOD: i64 = 10` with comment "10 seconds for devnet demo".
**Impact:** 10 seconds is not enough real time for a player to realize the AI ruled incorrectly, open the app, and submit a dispute. On mainnet this makes the dispute flow functionally useless — finalize will always race the player.
**Fix:** Set to `300` (5 min) for mainnet. Also update: `backend/src/services/solana.service.ts:281` (`+ 10` → `+ 300`), `backend/src/services/finalizer.service.ts:41` (`POLL_INTERVAL = 10_000` → probably `60_000` — poll every minute when challenge is 5 min), `mobile/src/config/index.ts:40` (`GAME_CONFIG.CHALLENGE_PERIOD = 10` → `300`).

### C-4. Single-step authority transfer
**File:** `contracts/programs/seek-protocol/src/lib.rs:1161`
**Current:** `transfer_authority` sets `global_state.authority = new_authority` immediately in one signature.
**Impact:** A typo or clipboard-swap attack in `new_authority` = permanent protocol loss. No way to recover. README claims "two-step: propose + accept" but the code is one-step.
**Fix:** Add `pending_authority: Pubkey` field on GlobalState. `transfer_authority` sets pending, new `accept_authority` instruction requires signature from the pending address to finalize. Optional: add `cancel_authority_transfer` for the current authority to rescind before acceptance.

### C-5. Mainnet production URL not configured
**File:** `mobile/src/config/index.ts:8,14`
**Current:** `NGROK_URL = null` → falls back to `PROD_URL = 'https://api.seek.app/api'`. The domain `seek.app` is not purchased/configured (verify).
**Impact:** Every mainnet user will hit DNS failure on API calls; app is bricked.
**Fix:** Register production domain (seek.app if available, else pick), point to Railway/Fly.io backend, set SSL cert, update `PROD_URL`. Also: remove `NGROK_URL` from release builds via build-time flag.

## SECTION 2 — HIGH (should fix before mainnet traffic)

### H-1. In-memory state loss on restart
**File:** `backend/src/services/bounty.service.ts:6–17`, `finalizer.service.ts:35`
**Maps lost on restart:**
- `activeBounties` → all in-flight bounty records
- `bountyByPlayer` → wallet→bountyId index
- `missionSecrets` → mission_id + salt for commit-reveal
- `preparedBounties` → pre-on-chain bounty state
- `walletLocks`, `bountyLocks` → mutex state
- `pendingFinalizations` → finalize queue

**Impact:** Every backend restart (deploy, crash, scale event) orphans every in-flight bounty. Player cannot submit photo because backend lost the mission secrets → `cancel_bounty` eventually refunds, but UX is broken for the 1-hour grace period. Also: horizontal scale impossible (two instances have independent Maps, race conditions).
**Fix:** Move all 7 Maps to Redis (Upstash serverless, $0 at low volume). Keys: `bounty:{id}`, `bounty:player:{wallet}`, `mission:secrets:{bountyId}`, `bounty:prepared:{bountyPda}`, `lock:wallet:{wallet}` (SETNX with TTL), `lock:bounty:{id}` (SETNX with TTL), `finalizer:queue` (sorted set by challengeEndsAt). Add optional Postgres later for analytics/history.

### H-2. Authority private key in .env
**File:** `backend/src/config/index.ts:16`, `.env`
**Current:** `AUTHORITY_PRIVATE_KEY` is a base58 string loaded into memory via dotenv.
**Impact:** Any backend compromise = full protocol compromise. On mainnet, this keypair is the sole signer for `reveal_mission`, `propose_resolution`, `resolve_dispute`, `withdraw_treasury`, `transfer_authority`, and `fund_house`. Leaking it drains the treasury.
**Fix paths (ordered by effort):**
- Minimum: Railway/Fly.io secrets manager (encrypted at rest, audit logs).
- Better: split authority. Hot keypair only has scope for `reveal_mission` + `propose_resolution` (the time-sensitive reveal flow). A separate cold wallet / Squads multisig gets `withdraw_treasury`, `transfer_authority`, `resolve_dispute`. Requires contract change: `hot_authority` vs `cold_authority` fields.
- Best: KMS-backed signer (AWS KMS, Google Cloud HSM) for hot authority, Squads multisig for cold.

### H-3. Program upgrade authority = single keypair
**Deployment-level, not in code.**
**Current:** Whoever ran `anchor deploy` is the program upgrade authority.
**Impact:** A single key can push malicious program code. No review, no timelock.
**Fix:** After mainnet deploy, transfer upgrade authority to a Squads v4 multisig. Use Squads Safe Authority Transfer (SAT) flow. Optionally add a Squads time lock (24h) on program upgrades so exploits can be surfaced before they land.

### H-4. Finalizer in-memory queue is stateless across restarts
**File:** `backend/src/services/finalizer.service.ts:35`
**Impact:** If backend crashes between `propose_resolution` and `finalize_bounty`, the bounty is stuck in `ChallengeWon/ChallengeLost` forever until `cancel_bounty` grace period expires (1 hour) — and even then cancel only works if still in Pending/Submitted state, so a ChallengeWon stuck bounty is permanently stuck.
**Fix:** Redis sorted set (`ZADD finalizer:queue <challengeEndsAt> <bountyPda>`). Worker pops with `ZRANGEBYSCORE ... 0 <now>`. On success: ZREM. On restart: picks up where it left off. Also: add a cron-scan that lists all Bounty accounts in `ChallengeWon/ChallengeLost` status on-chain and reconciles with queue (belt-and-suspenders).

### H-5. No Sentry / APM / structured logging
**Files:** `backend/src/index.ts`, mobile app
**Current:** `console.log` everywhere, `console.error` for errors. No correlation IDs. No error aggregation. No crash reports on mobile.
**Impact:** First production bug = flying blind. Have to ask users to screenshot error modals. Crashes on device never surface.
**Fix:** Add @sentry/node to backend (5 min setup). Add @sentry/react-native to mobile (~30 min with symbolication). Free tier covers launch. Also: add request IDs via middleware, pass through to Sentry tags.

### H-6. APK size 138MB — likely unminified, likely bundled dev tools
**File:** `mobile/android/app/build.gradle:82`
**Current:** `enableMinifyInReleaseBuilds = false`. No R8/Proguard. Hermes enabled. expo-dev-client included.
**Impact:** Users wait 2+ minutes to download on cellular. Also 138MB is larger than most apps in dApp Store.
**Fix:** Flip `enableMinifyInReleaseBuilds = true`. Add Proguard rules for: react-native, expo, @solana/web3.js, @wallet-ui (may need `-keep class` rules — test release build). Remove `expo-dev-client` dep for release builds or gate behind `__DEV__`. Run `./gradlew assembleRelease --scan` to see asset breakdown. Expected result: ~40-60MB.

### H-7. AndroidManifest allows cleartext HTTP
**File:** `mobile/android/app/src/main/AndroidManifest.xml`
**Current:** `android:usesCleartextTraffic="true"`.
**Impact:** App will send requests over HTTP if config points there. Allows MITM on tunnel/staging endpoints. Should be false for production.
**Fix:** Remove attribute (defaults to false). If specific dev endpoints need cleartext, use a `network_security_config.xml` to scope to `debug` buildType only.

### H-8. AndroidManifest over-permissioned
**File:** same manifest
**Current:** Requests `WRITE_EXTERNAL_STORAGE`, `READ_EXTERNAL_STORAGE`, `READ_MEDIA_VIDEO`, `READ_MEDIA_AUDIO`, `RECORD_AUDIO`, `MODIFY_AUDIO_SETTINGS`, `SYSTEM_ALERT_WINDOW`, `allowBackup=true`, `requestLegacyExternalStorage=true`.
**Impact:** dApp Store reviewers flag over-permissioned apps. Extra attack surface. `RECORD_AUDIO` for a photo-based game is a bad look.
**Fix:** Audit per-feature. Keep: `CAMERA`, `INTERNET`, `VIBRATE`. Consider keeping: `READ_MEDIA_IMAGES` (if photo picker ever used). Remove: `RECORD_AUDIO`, `MODIFY_AUDIO_SETTINGS`, `SYSTEM_ALERT_WINDOW`, `WRITE_EXTERNAL_STORAGE`, `READ_EXTERNAL_STORAGE`, `requestLegacyExternalStorage`, `allowBackup=true`. Expo modules may re-add some on install — verify after.

### H-9. No release keystore configured
**File:** `mobile/android/app/build.gradle`
**Current:** `signingConfigs` block present but likely empty/debug.
**Impact:** **Solana dApp Store REJECTS debug builds outright** per docs.solanamobile.com. Must be signed with a NEW keystore (not your Google Play Store key). Lost keystore = cannot update the app ever.
**Fix:** Generate 2048 RSA keystore (`keytool -genkeypair`), back up encrypted + print paper backup. Wire Gradle to read from env vars (`SEEK_KEYSTORE_PATH`, `SEEK_KEYSTORE_PASSWORD`, `SEEK_KEY_ALIAS`, `SEEK_KEY_PASSWORD`). Never commit the keystore.

### H-10. Cloudflare tunnel ≠ production infra
**Current:** `backend` is reachable via ephemeral Cloudflare tunnel URL, recorded in `mobile/src/config/index.ts` before each build. Per commit `8425a52`, tunnel URL is now reset on each commit.
**Impact:** Obviously not a production setup. Tunnel URL regenerates, tunnel process can die, no SLA, no scale. Also: any mobile build against a tunnel URL is a time-bomb once the tunnel is down.
**Fix:** Move backend to Railway (recommended) or Fly.io. Add managed Redis (Upstash). Set custom domain + SSL. Build mobile APK against production URL only.

**Impact:** LOW — recoverable, but worth adding idempotency: check if bounty is already `Submitted` before calling reveal again.

## SECTION 5 — POSITIVE OBSERVATIONS

1. **Architecture is genuinely clean.** Commit-reveal + optimistic resolution + dispute window is more sophisticated than most hackathon contracts. The pattern scales well.
2. **Arithmetic is checked everywhere.** `checked_add` / `checked_mul` / `checked_div` / `saturating_sub` are used consistently. No integer overflow risk found.
3. **Account constraints are thorough.** Mint checks, owner checks, duplicate-account guards in `finalize_bounty` (lines 1403–1405) — good paranoia.
4. **`close_bounty` correctly uses Anchor's `close = player`.** Rent reclaim is clean.
5. **Commit-reveal implementation is correct** (SHA-256 of `mission_id || salt`, verified on-chain on reveal).
6. **Cancel flow is well-designed** — 1h grace period + proper state check + balance tracking via `saturating_sub`.
7. **Backend EXIF + AI dual-check is sound.** Claude Vision + metadata cross-reference is stronger than single-gate approaches.
8. **Backend locks (wallet + bounty) catch double-submit race.** Per-wallet + per-bounty mutex pattern.
9. **`verifyTransaction` is called before creating bounty state** (bounty.routes.ts:171–180) — closes the impersonation hole from prior audit.
10. **Finalizer queue retries 10x before giving up** — matches a realistic transient-failure model.
11. **Zod schemas validate all request bodies** — base58 regex for wallet addresses is correct.

## SECTION 9 — OPEN DECISIONS FOR USER

1. **VRF provider: Switchboard On-Demand vs Orao?** Switchboard is the community default with cheaper per-request fees (~0.002 SOL) and Anchor-friendly SDK. Orao is simpler to integrate but costs more and less battle-tested at scale. Recommend **Switchboard**.
2. **Hosting: Railway vs Fly.io vs AWS?** Railway recommended — simplest, has Upstash Redis addon, env-vars UI, free trial + pay-as-you-go. Fly.io for multi-region + Docker control.
3. **Persistence: Redis-only or Redis + Postgres?** Recommend Redis-only at launch. Add Postgres later for analytics/leaderboard/history.
4. **External audit firm + budget?** Neodyme ~$40k / OtterSec ~$50k / Accretion ~$25k for ~2 weeks typical.
5. **Squads multisig signers?** Need 3+ keypairs. Recommend: founder + co-founder + one cold hardware wallet as 2-of-3. Or 2-of-4 if adding a trusted third party.
